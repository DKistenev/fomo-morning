Introducing Claude Sonnet 5

Новая Sonnet-модель для агентных задач: планы, браузер, терминал и выбор уровня effort — с опубликованными оговорками по безопасности и цене.

Контекст

Агентный кодинг упирается не только в качество модели, но и в стоимость длинных запусков, проверку результата и риски инструментального доступа. Поэтому практическая ценность релиза — не в общем заявлении о «фронтирности», а в доступном API, режимах усилия и раскрытых оценках безопасности.

Как устроено
  • Anthropic заявляет, что Sonnet 5 умеет планировать, использовать браузер и терминал; модель доступна в Claude Code и через API как `claude-sonnet-5`.
  • Пост сравнивает Sonnet 5 с Sonnet 4.6 и Opus 4.8 на BrowseComp и OSWorld-Verified и описывает выбор уровня effort как компромисс цены и качества.
  • На старте указана цена $2 за миллион входных и $10 за миллион выходных токенов до 31 августа 2026 года; затем — $3 и $15 соответственно.
  • Для модели включены cyber safeguards; в опубликованных оценках она не смогла создать полноценный рабочий эксплойт для Firefox, хотя имела отдельные частичные успехи.
Ограничение

Это данные и оценки самого поставщика; конкретная производительность зависит от задачи, инструментария, лимитов и выбранного effort. Обновлённый токенизатор может менять число токенов для того же входа.

Практический вывод

Проведите короткий A/B на своей агентной цепочке: измерьте не только успешность, но и стоимость полного прогона, число вмешательств человека и долю безопасно заблокированных действий.

More details on Fable 5’s cyber safeguards and our jailbreak framework

Подробная схема классификаторов для киберзапросов и ранний проект общей шкалы тяжести jailbreak-обходов.

Контекст

Одна и та же модель может помогать защитнику искать уязвимости и помогать атакующему. Простое правило «запретить кибербезопасность» ломает легитимную защитную работу, а разрешение без границ повышает риск злоупотреблений; нужен явный язык категорий и режимов реакции.

Как устроено
  • Anthropic описывает четыре категории: запрещённое использование, high-risk dual use, low-risk dual use и benign use; для них предусмотрены блокирование, мониторинг или разрешение с мониторингом.
  • Классификаторы — только один слой: дополнительно названы контроль доступа, обучение безопасности и офлайн-мониторинг.
  • «Safety margin» намеренно блокирует часть доброкачественных и низкорисковых запросов, чтобы снизить шанс пропустить вредные; это означает больше ложных срабатываний.
  • Проект jailbreak severity framework предназначен для более согласованного описания того, какие обходы разблокируют опасное поведение.
Ограничение

Это ранняя позиция одного разработчика, а границы классификаторов и примеры могут меняться. Не следует трактовать такую категоризацию как универсальный стандарт или как гарантию отсутствия обходов.

Практический вывод

Для собственных AI-инструментов заведите журнал отказов и разрешений: разделяйте вредные, высокорисковые dual-use, низкорисковые и обычные операции — так можно измерять ложные блокировки вместо спорить о них абстрактно.

SWE-Pruner Pro: The Coder LLM Already Knows What to Prune

Работа предлагает вырезать строки tool output прямо по внутренним представлениям кодового агента, экономя контекст без отдельного классификатора кода.

Контекст

В длинных агентных сессиях результаты команд, логи и файлы быстро заполняют контекст. Отдельный pruner добавляет систему и задержку; интересная альтернатива — извлечь оценку релевантности из самого агента, который уже читает этот вывод.

Как устроено
  • Авторы предлагают маленькую head, которая по внутренним представлениям агента ставит каждой строке tool output метку keep или prune; используется length-aware embedding для длины вывода.
  • В абстракте заявлены тесты на двух open-weight backbones и четырёх многоходовых бенчмарках.
  • Заявленная экономия достигает 39% токенов prompt и completion при сохранении качества, с ограниченным inference overhead.
  • Для MiMo-V2-Flash авторы сообщают +3,8 п.п. к SWE-Bench Verified resolve rate и +2,2 п.п. к точности на long-context Oolong.
Ограничение

Это свежая препринт-работа: заявленные результаты нужно перепроверить на вашем агенте, репозитории и типах логов. Pruning может удалить редкий, но критичный диагностический фрагмент.

Практический вывод

До добавления сложной памяти попробуйте измеряемый line-level pruning: сохраняйте удалённый вывод отдельно, логируйте причину удаления и сравните success rate, токены и время восстановления ошибок.

Environment-free Synthetic Data Generation for API-Calling Agents

Метод синтезирует траектории API-агентов по одной спецификации API, заменяя полноценное исполняемое окружение LLM-симулятором состояния.

Контекст

Собрать реальные агентные траектории дорого: нужны API, бэкенды, состояние и данные. Но обучать вызовам API на одном текстовом описании опасно, если симулятор будет непоследовательным или «научит» агента несуществующему поведению.

Как устроено
  • Авторы начинают с API specifications: LLM генерирует разнообразные решаемые задачи, затем teacher-agent решает их против LLM-симулятора, который формирует согласованные API-ответы с учётом истории.
  • Отдельный LLM judge фильтрует траектории перед включением в датасет.
  • Оценка заявлена на AppWorld и OfficeBench, где есть как информационный поиск, так и изменяющие состояние задачи.
  • Авторы сообщают, что fine-tuning на синтетических данных дал значимые улучшения, не требуя исполняемого окружения.
Ограничение

«Значимые улучшения» в абстракте не заменяют проверку переноса на реальные API, ошибки и пограничные состояния. Симулятор может закрепить неверную семантику или недооценить сбои сети и авторизации.

Практический вывод

Для нового internal API сначала создайте набор спецификаций и тестовых инвариантов, затем сравните агента, обученного на симуляции, с реальным sandbox до любого доступа к production.

Why Some AI Workflows Stick—And Others Don’t

Разбор того, почему эффектный AI-workflow может оказаться лишней очередью: важны реальная частота проблемы, поддержка привычки и отдача быстрее цены обслуживания.

Контекст

Построить персональную автоматизацию теперь легко, поэтому команды часто начинают с демо, а не с болевой точки. В результате агенты добавляют сообщения, настройки и maintenance вместо того, чтобы убирать работу — особенно если система соревнуется с устойчивой привычкой пользователя.

Как устроено
  • Автор описывает Attention Desk, собранный с Codex для агрегации email, Slack, заметок встреч и обновлений, но впоследствии заброшенный как решение несуществующей для неё проблемы.
  • В качестве диагностических причин названы редкость задачи, отсутствие триггера и ситуация, когда output создаёт больше работы, чем экономит.
  • Статья ссылается на обзор 2024 года: медиана формирования привычки около двух месяцев, диапазон — от четырёх до 335 дней.
  • Также приводится метаанализ 2024 года из 642 тестов: ясные планы в конкретном контексте помогают, особенно когда мотивация уже высока.
Ограничение

Это практический эссе-разбор одного автора, а не универсальная модель внедрения. Поведенческие исследования в статье не тестируют именно AI-workflows.

Практический вывод

Перед недельным билдом спросите: какая конкретная работа выполняется часто, что станет триггером и сколько минут нужно сэкономить уже в первый день. Если ответа нет — сделайте маленький прототип, а не постоянную систему.

После того как агент написал код и проверки зелёные, следующий bottleneck — человеческая оценка интерфейса в работающем продукте.

Контекст

Тесты и review-агенты хорошо ловят формальные дефекты, но не определяют, «правильно ли ощущается» анимация, плотность экрана или взаимодействие в контексте продукта. Если считать green CI окончанием работы, агентная скорость превращается в поток функциональных, но безликих фич.

Как устроено
  • Автор описывает workflow /ce-polish: checkout ветки, запуск dev server в фоне и открытие приложения рядом с агентом.

Reverse-engineering is cheap now

Coding agents снижают стоимость проб и ошибок при автоматизации домашних устройств и недокументированных интерфейсов — но не устраняют стоимость будущей поддержки.

Контекст

Недокументированный API раньше был технически возможен, но экономически часто невыгоден: начальная разработка и неизбежный ремонт после изменений съедали ценность автоматизации. Агент удешевляет исследование, поэтому порог оправданности сдвигается, но хрупкость интеграции остаётся.

Как устроено
  • Автор приводит наблюдения о людях, которые применяют coding agents для reverse engineering и автоматизации домашних устройств.
  • Главный эффект сформулирован как снижение усилия на первый работающий вариант и на неудачные попытки.
  • Указанный прежний риск — недокументированные и нестабильные API могут измениться или сломаться.
  • По мнению автора, удешевление кода снижает психологическую цену выбросить или заново написать такую автоматизацию.
Ограничение

Это наблюдательная заметка с anecdotes, а не измерение надёжности или экономии времени. Дешевле написать не значит безопасно подключить устройство к сети или устойчиво поддерживать интеграцию.

Практический вывод

Для каждого agent-built интегратора заранее задайте бюджет поддержки: health-check, контрактные тесты на API и простую кнопку отключения. Тогда дешёвый эксперимент не станет дорогой скрытой зависимостью.

Claude Code uses Bun written in Rust now

Разбор установленного бинарника Claude Code показывает признаки встроенного Bun на Rust; заметка — полезный пример проверяемого изучения developer tooling.

Контекст

Инструменты AI-кодинга становятся частью локальной инженерной среды. Даже небольшая смена runtime может влиять на старт, совместимость и отладку, но слух о переписывании нельзя превращать в факт без воспроизводимой проверки бинарника и версии.

Как устроено
  • Автор проверил установленный Claude бинарник командой `strings ~/.local/bin/claude | grep -m1 'Bun v1'` и получил строку `Bun v1.4.0 (macOS arm64)`.
  • Вторая команда поиска путей Rust-файлов вернула 563 имён; в заметке приведены примеры `src/runtime/bake/...` и `src/bundler/bundle_v2.rs`.
  • Автор связывает наблюдение с заявлением Jarred Sumner о Rust-порте Bun и обновляет текст: версия доступна как Bun canary.
  • Предлагается также preload-скрипт `BUN_OPTIONS`, который печатает встроенную версию Bun при запуске `claude --version`.
Ограничение

Это независимая инспекция конкретной установки на macOS, а не официальная спецификация Claude Code. Строки и файлы в бинарнике не гарантируют одинаковый runtime на всех платформах и версиях.

Практический вывод

При необычной регрессии CLI сначала зафиксируйте бинарник, встроенный runtime и платформу простыми командами; это быстрее, чем строить выводы по changelog или слухам.

SQLite Query Explainer

Browser-инструмент выполняет SQLite-запрос и объясняет строки EXPLAIN QUERY PLAN и bytecode EXPLAIN понятным языком.

Контекст

LLM может быстро объяснить план запроса, но объяснение остаётся гипотезой, если не видно самого плана и неясно, где выполняются данные. Полезный учебный инструмент должен держать рядом запрос, реальный вывод движка и интерпретацию, а не заменять одно другим.

Как устроено
  • Инструмент запускает SQL-запросы против SQLite в браузере и аннотирует каждую строку `EXPLAIN QUERY PLAN` и низкоуровневого bytecode `EXPLAIN`.
  • По описанию автора, SQLite работает в Python через Pyodide в WebAssembly, то есть исполнение происходит в браузере.
  • Интерпретация предназначена для того, чтобы показать, что делает query planner и виртуальная машина SQLite.
  • Автор прямо предупреждает, что сам не может полностью верифицировать объяснения инструмента и советует пользоваться осторожно.
Ограничение

Объяснения, сгенерированные AI, не являются авторитетным источником для оптимизации production-запросов; сверяйте их с документацией SQLite, измерениями и настоящими планами вашей версии БД.

Практический вывод

Используйте такой инструмент как учебный слой: в code review прикладывайте исходный SQL и реальный `EXPLAIN QUERY PLAN`, а рекомендацию агента принимайте только после измерения на репрезентативных данных.
